Digital Signature Schemes

Remember our friends Alice, Bob, and Eve? Now the plot thickens. Not only is Eve eavesdropping on the messages between Alice and Bob, but she is also trying to inject her own messages into their correspondence.

Alice receives a message from Bob that says he is breaking up with her. How does Alice know that the message is really from Bob and isn't a fake from Eve?

Cryptography gives us a useful tool to help Alice and Bob thwart Eve. This tool is called a digital signature scheme.

In a digital signature scheme, Bob has two keys, a secret key, SK, and a public key, PK.

Here, the secret key is the "secret sauce" that allows him–and only him–to sign a message, and the public key is what allows other people to verify Bob's signature on a message.

A digital signature scheme consists of three algorithms:

Key Generation: where a secret key, SK, and a public key, PK, are generated


Sign: on input a secret key, SK, and a message, m, generate a signature, denoted σ


Verify: on input the public key, PK, the message, m, and the purported signature σ, verify that σ is valid; accept if it is, reject if it is not.


And in order for a digital signature scheme to work, it must be:

Correct: Verify(PK,m,σ) = accept whenever σ = Sign(SK,m), where SK corresponds to PK.


Secure (Unforgeable): no matter how long Eve eavesdrops on the channel and observes Bob sign various messages, she still cannot spoof Bob's signature.

So let’s see how this all plays out in the drama between Alice and Bob. How should they ensure that they can communicate securely and reliably?

First of all, Alice and Bob need to set up their keys, both for encryption and for digital signatures. Let PKenc denote the public key for encryption, and PKsign denote the public key for signing; similarly, let SKenc and SKsign denote their secret variants.

Then, they need to give each other the public portions of these keys.

Next, if Bob wants to send a message to Alice, he should use his SKsign to sign it and get σ, and then use Alice's PKenc to encrypt m and σ.

So now, if Alice receives a message that purports to be from Bob, all she needs to do to check that it is indeed from Bob (and not a spoof from Eve) by verifying the signature on it.

If the signature does not verify, then Bob didn't break up with her after all, and they have more exciting adventures in cryptography in store.

 

Hash Functions and How They Are Used; Certification

So now we know that in order to make it possible to digitally sign messages, the signer needs to have set up his/her key pair (PK, SK), and the recipient needs to know the signer's PK.

But what happens during the signing algorithm? There are many different cryptographic algorithms, but the most common scenario is as follows.

If Bob wants to send a message to Alice, first, Bob settles on the message, M, that he wants to sign. Potentially, M can be a very large document.

Now, you’ll notice we’re using a capital “M” here. This is in contrast to the lowercase “m” we used earlier, and you’ll see why in a moment.

Next, Bob "hashes" M to a relatively short string, typically just 256 bits. For this, Bob uses a cryptographic hash function, H. Right now, how H works is not important. But what is important to know is that H should be collision-resistant; that is, it should be nearly impossible to find two different messages, M1 and M2, that will hash to the same value.

The resulting short string, m = H(M), where m is 256 bits, is what is actually signed. That way, even if the original document, M, is very large, the signature is still small.

Also, using H, we can make sure that, even if Bob’s signing PK is quite large, Alice need not store it. All she needs to remember is pk = H(PK), which is just a 256-bit string. As long as Bob includes his PK every time he sends her a signed message, she can verify that PK is correct by checking that pk = H(PK).

So how do Alice and Bob use a hash function, H? Alice stores pk = H(PK) for Bob, and receives from Bob the message, M, his public key, PK, and the signature σ = Sign(SK,H(M)). Here is what she must do to verify Bob’s signature:

  1. Check that pk = H(PK)
  1. Check Verify(PK,H(M),σ) = accept

But maybe Alice and Bob never met before, so Alice does not know Bob’s pk = H(PK), or indeed anything about his PK. She receives (PK,M,σ) over the network. How does she verify that, in fact, it is from Bob and not from someone else?

If she is tricked into thinking that a PK that Eve sent to fool Alice is the correct PK for Bob, it can have catastrophic consequences: Eve will be able to both read the messages Alice thinks she is sending to Bob in confidence and spoof messages from Bob to Alice.

This is the problem of certification. However, there are several solutions:

  1. Self-certification: In this scenario, Bob gives Alice his PK in person, so that she knows it's his because she received it from him via a channel that is already authenticated.

  1. Peer certification: Here, Alice and Bob use a mutual friend, Fred, whom Alice trusts, and whose PK Alice already knows. If Fred has signed Bob's PK – a signature that certifies the validity of a public key is called a certificate – then Alice can be reasonably sure that it is correct.

  1. Certification authority: This method uses a certification authority that is globally trusted. One example could be Brown University. In this scenario, Brown University certifies that Bob’s PK is, indeed, his correct PK, and not one spoofed by Eve.
Randomness

Up until now, we pictured a key generation algorithm like this:

But what is inside this box that makes Alice's and Bob's keys safe from Eve?  What stops Eve from running exactly the same key generation algorithm to obtain exactly the same keys?

The magic sauce is randomness.

How do we generate a truly random sequence of bits? One time-tested method is to simply flip a coin.

Half the time it comes up heads, half the time tails. If you need a long truly random string of bits, just flip a coin enough times.

So now our updated view of a key generation algorithm is this:

So even though Eve is running the same exact algorithm as Bob, unless she is lucky enough to pick the same sequence of bits to go into it, her output will be different from Bob's.

Of course, in reality, there is no coin inside your computer that will flip bits for you. Instead, your computer generates a sequence of bits that might or might not be good enough. So what are some common techniques for generating a random sequence inside your computer?

Randomness, also known as entropy, may come from a user's keystrokes or mouse movement.

It can also originate from the timing of various messages arriving over the network, or internal processes handing over control to each other.

Sometimes, instead of looking for fresh entropy from its physical environment, your operating system might produce bits algorithmically, using so-called pseudorandom generators.

What can go wrong in the process?  What happens if, instead of being truly random, the bits that go into the key generation algorithm are somehow skewed?

One example is if the coin is skewed, and it always comes up heads. Eve can then easily generate the exact same key as Bob.

Or, sometimes, even though he might have some true randomness to work with, his randomness is just not good enough for the security proof of the cryptosystem to hold. Whatever proof of security we have for the original algorithms that Alice and Bob were supposed to be using no longer carries over because the proof of security assumed that the randomness was perfect.

So now, because the proof doesn’t hold anymore, we are vulnerable to cryptanalysis.

Therefore, before you generate your keypairs, you must make sure that you have good randomness.

The certification authority that issued this certificate is Symantec, which is a company that, among other things, is in the business of ensuring that it correctly associates public keys with the owners of the corresponding domains. Symantec's public key is hard-wired into your browser, which is why your browser can verify Amazon's certificate without accessing any other web resources. In addition to Symantec's public key, other popular certification authorities' public keys are also known to your browser (for example, that of Verisign). Should any of those certification authorities get hacked, the security of your connections to services such as amazon can be compromised; this has been a source of serious vulnerabilities in the past. Luckily, updating your browser software and a judicious use of expiration dates mitigates the problem to a great extent.

If you look at more details, you will also see the SHA1 and the MD5 “fingerprint” of Amazon.com’s public key. SHA1 and MD5 are both cryptographic hash functions, and these fingerprints are just hashes of the full public key.

When you want to buy something from amazon.com, you must first establish a secure connection with it. So how do you do it?

First, some terminology: in this transaction, you – or, rather, your browser – play the role of the client, while amazon.com is called the server.

Even before your transaction begins, the server has established a public key, and it has been certified by one of the certification authorities your browser trusts.

So when you talk to amazon.com, you can already make sure that you are talking to the correct server: it can use digital signatures to sign whatever messages it sends to you.

You can also send it your own public key so that your connection can be encrypted. However, the more common thing to do, instead of sending a public key, is to pick a symmetric key, K, and send it to the server, encrypted under the server's public key. The client and server use K to encrypt and authenticate their communication throughout the session. This key is called "the session key."

There is, however, a vulnerability to this approach. If Amazon's secret key is ever compromised, K can be decrypted. So if you sent your credit card info encrypted under K, a hacker who managed to eavesdrop it today and break into amazon.com sometime later can decrypt it.

The more desirable (but less commonly deployed) scenario is slightly different. Instead of picking K and sending it in encrypted form, the client and server generate it over an insecure network using a key exchange protocol, typically the Diffie-Hellman key exchange.

We’ll go more in-depth into this method later, but the advantage of this method is that, in this case, the session key disappears after the session dies out: it was never sent in encrypted form.

Any time you see "https" at the start of a URL in your browser, it means that your browser has connected to a server, verified that it has connected to the correct server (using signatures), established a session key, K, with the server, and now encrypts and authenticates everything sent in this session using K.
