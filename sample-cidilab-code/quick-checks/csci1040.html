The symmetric-key analogue of a digital signature scheme is called a message-authentication code (MAC). Here, Alice and Bob share a symmetric key K; Alice uses K in order to compute an authentication tag t on her message M to Bob, so t = MAC(K,M); when Bob receives M and t from Alice, he checks whether t = MAC(K,M).

In other words, a message authentication code consists of two algorithms: a key generation algorithm KeyGen and an algorithm MAC. KeyGen is a randomized algorithm that outputs a key K, and MAC takes an input a key K and a message M and outputs a tag t.

It is actually desirable that any sufficiently long string (sufficiently long typically means 160 bits) can work as the key K, so the key generation algorithm just consists of picking a 160-bit string K uniformly at random.

Building on our definition of security for a digital signature scheme, let us give one for MACs.

 

Definition of unforgeability for signatures: Let a key K be chosen according to the KeyGen algorithm. Suppose an adversary Eve has the following available to her:
  • The ability to obtain tags t = MAC(K,M) for messages M of her choice;
  • All the computational resources that can reasonably be available to a powerful adversary.

A message authentication code is secure if Eve’s likelihood to produce a tag t for a message M that she has not explicitly obtained is infinitesimally small.

 

In Module 2, when we talked about how CBC mode was in practice, a message authentication code was a missing piece. Recall that Alice used public-key encryption to send Bob a symmetric key K, which she then used to encrypt her actual message using symmetric encryption, and to authenticate the resulting ciphertext.

How are MACs constructed? There are many candidate constructions in the literature. The one that’s most frequently used in practice is called “Hash-based MAC,” or HMAC for short, and it was designed by three cryptographers: Mihir Bellare (of the RSA-OAEP and FDH-RSA constructions), Ran Canetti and Hugo Krawczyk. It is very simple: HMAC(K,M) = H((K ⊕ A) ⚬ H((K ⊕ B) ⚬ M)), where ⊕ denotes the exclusive-or operation, ⚬ denotes concatenation, and A and B are two fixed strings. It is easy to see that it is secure in the random-oracle model.

 

  Self Check

Why intuitively is HMAC secure in the random-oracle model?

Answer:

To compute HMAC(K,M), we need to be able to compute H((K ⊕ A) ⚬ H((K ⊕ B) ⚬ M)). In the random-oracle model, we can only compute H(x) by explicit queries, and so we would need to know K in order to query the hash function H for the values (K ⊕ B) ⚬ M and (K ⊕ A) ⚬ H((K ⊕ B) ⚬ M).

The beauty of HMAC is that it was shown to be secure even if H is not quite random-oracle-like, but is constructed in a way some frequently used candidates for cryptographic hash functions were constructed, such as MD5 and SHA-1. Interestingly, even though, as we just saw, MD5 is badly broken, instantiating HMAC with MD5 seems to still withstand cryptanalysis.
